# How to: extend the language

This document is the recipe for adding a new construct to the `.tsn` DSL (a new
top-level `defsomething`, a new term inside `defnetwork`, a new expression form,
etc.). It is written to be followed by either a human or an LLM: read it, then
implement the construct stage by stage.

---

## General principle

The language is a **pipeline**. Source text flows through fixed stages, and each
stage has exactly one home file:

```
source text
   â”‚  Lezer grammar            src/data-network/grammar.grammar   (+ generated parser.js)
   â–Ľ
parse tree (CST)
   â”‚  collector                src/data-network/tree-to-network.ts
   â–Ľ
AST  (ProgramAST)              src/data-network/types.ts
   â”‚  type-checker             src/data-network/type-checker.ts
   â–Ľ
checked AST
   â”‚  compiler / runtime       src/sandbox/jsgen/*, src/sandbox/*-runtime.ts
   â–Ľ
runnable network
```

**Adding a construct means threading it through these stages in order.** You
rarely need every stage at once â€” how far you go depends on what the construct
*is*:

- A **declaration** that only needs to be parsed and represented (a record, an
  enum, a parameter) stops at the AST. Stages 1â€“3.
- A **callable** that participates at runtime (a function, an LLM function, a
  grammar) continues into the type-checker and the sandbox/registry. Stages 1â€“6.

Decide the target depth first, and say so â€” it is fine and normal to land a
construct at "parse + AST" and defer the runtime wiring to a later change.

Two project rules govern the whole process:

- **Tests first (TDD).** Write the parse tests before touching the grammar. See
  the test methodology in `CLAUDE.md` (capabilities / invariants / negatives /
  units).
- **The suite stays green.** Run `npm test` (it regenerates the parser first â€”
  never run `npx jest` raw, it uses a stale parser). Run `npx tsc --noEmit`
  whenever you change types.

---

## The stages, as a checklist

1. **Write the parse tests** â€” `tests/data-network/parse-program.test.ts`.
   Encode the target syntax and the AST you expect, plus at least one negative
   (malformed input must be a syntax error).
2. **Grammar** â€” `src/data-network/grammar.grammar`. Add the rule, register it in
   the `Definition` alternation (for a top-level construct), and declare any new
   keywords. Regenerated by `npm test`; the generated `parser.js` /
   `parser.terms.js` are **not** git-tracked.
3. **AST type** â€” `src/data-network/types.ts`. Add the `XxxAST` type and, for a
   top-level construct, a field on `ProgramAST`.
4. **Collector** â€” `src/data-network/tree-to-network.ts`. Walk the parse tree
   into the AST type, and dispatch to it from the top-level walk.
5. **Fix `ProgramAST` literals.** A few tests build a `ProgramAST` by hand; a new
   required field breaks them until you add it. `grep` for them.
6. **(Callables only)** type-checker, then sandbox/registry
   (`src/sandbox/jsgen/registry.ts`), then a runnable `examples/*.tsn`, then a
   note in `README.md`.

Verify continuously: `npm test`, and `npx tsc --noEmit` after type changes.

## A worked example: `defparameter`

`defparameter` is a named, overridable input with an optional default. Target
syntax:

```
defparameter myArticle
  type: Text?;
  value:
    """ ... """;
end
```

It is a *declaration*, so it stopped at stage 5. Here is what each stage looked
like â€” illustrative fragments, not the full code.

**1. Tests first.** A `describe("parseProgram: defparameter", ...)` block that
parses the snippet above and asserts the resulting AST node, plus a no-`value`
case, an opacity case (punctuation inside `value` is preserved), and a negative
(missing `type:` throws a syntax error).

**2. Grammar.** Add the rule and slot it into the alternation:

```
Definition { â€¦ | ParameterDef }

ParameterDef { Defparameter Name TypeClause ValueClause? End }
TypeClause  { Type ":" Name ";" }
ValueClause { Value ":" PromptString ";" }
```

Keywords are declared at the bottom of the grammar. There are **two kinds**, and
choosing correctly matters:

- `kw<"...">` (`@specialize`) â€” a *hard* keyword, reserved everywhere. Use it for
  construct names like `defparameter` that should never be identifiers.
- `ckw<"...">` (`@extend`) â€” a *contextual* keyword, recognised only where the
  grammar expects it and still usable as an ordinary `Name` elsewhere. Use it for
  common words (here `type`, `value`) so you don't steal them as field/cell names.

The triple-quoted `value:` body reuses the existing `PromptString` external
tokenizer, which swallows `"""â€¦"""` verbatim â€” the right move for any opaque blob
(prompts, grammar source, free text).

**3. AST type.** A node type plus a `ProgramAST` field:

```ts
export type ParameterAST = {
  kind: "parameter";
  name: string;
  type: TypeRef;      // reuse existing TypeRef â€” don't invent a parallel shape
  value?: string;     // optional: absent default == Nothing
};

export type ProgramAST = { â€¦; parameters: ParameterAST[] };
```

**4. Collector.** The collector is a cursor walk over the parse tree. The idiom
throughout `tree-to-network.ts`: `firstChild()` to descend, `nextSibling()` to
advance, `parent()` to come back up â€” always restore the cursor to where you
entered. Match on `cursor.name` (the grammar rule names), and slice source text
with `slice(cursor.from, cursor.to)`. Then dispatch from the top-level walk:

```ts
else if (cn() === "ParameterDef") parameters.push(collectParameterDef());
```

â€¦and include the new array in the returned `ProgramAST`.

**5. Fix literals.** Two test files build a `ProgramAST` inline; each needed
`parameters: []` added.

That was the whole change for a declaration. A callable would continue: teach
`type-checker.ts` about the new node, register a runtime implementation in
`src/sandbox/jsgen/registry.ts`, add an `examples/*.tsn`, and document it in
`README.md`.

## Conventions and gotchas

- **Reuse the shared shapes.** `TypeRef` (scalar / vector) and the `FnSignature`
  grammar rule + `collectFnSignature()` collector are shared across `defn`,
  `defpredicate`, `defllmfn`, and `defgrammar`. If your construct has a type or a
  signature, reuse these rather than inventing a parallel form â€” uniform syntax
  is a design goal.
- **`kw` vs `ckw`.** Reserving a common word as a hard keyword silently breaks
  programs that use it as a name. Prefer `ckw<>` unless the word is unmistakably
  a keyword.
- **Generated parser is not tracked.** `parser.js` / `parser.terms.js` are
  produced by `npm run generate` (and by `npm test`). Don't hand-edit or commit
  them.
- **Cursor discipline.** Every `firstChild()` that succeeds must be balanced by a
  `parent()` before the enclosing loop advances, or the walk desyncs. Anonymous
  tokens (`":"`, `";"`) appear as nodes â€” step over them with `nextSibling()`.
- **Run the right command.** `npm test` (regenerates the parser); `npm run
  test:all` to include the slow subprocess-based script tests; `npx tsc --noEmit`
  after any type change.
