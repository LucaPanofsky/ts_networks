Architecture Decision Record · ADR-001

Deploying Gavagai?

How we take the containerized authoring agent from a single-conversation box to an organization-internal tool that a small team can log into — with the API key, multi-user isolation, and persistence questions all settled.

01

Context & scope

Gavagai is a containerized Claude Code instance that authors .tsn programs using the language's own runtime — read-only mounted, so it can use the language but not change it. Today it runs as one container = one conversation: the Anthropic API key arrives through an environment variable at boot, and there is exactly one user.

Terminology What we mean by "container"

Throughout this record, container means an OS-level container — a single image, run as one isolated process environment with its own filesystem, users, and network view, sharing the host kernel rather than booting a full virtual machine. Think of it as a process in a sealed box: cheap to start and throw away, but with a real boundary the kernel enforces.

It is also our unit of deployment. When the design says it "spawns a process per user" or "stays single-container", it means everything happens inside this one box — the same box the org runs today, not a fleet.

We want to deploy it for real, internal use: an organization stands it up as a productivity tool — parse this text, extract that table, author this extractor — and a handful of employees log in and use it. This record fixes the deployment model and the three questions that come with it: the API key, multi-user isolation, and persistence.

This is a proof of concept for at most ~10 users. That number is a design input, not a footnote: it lets us choose the simple correct answer (one container, one key, OS-level isolation) over the scalable one (per-session containers, an orchestrator, autoscaling). We design it well; we do not build for a million users we do not have. Where that scale ceiling changes a choice, it is called out explicitly.

Guiding principle — SDK first

We build on the Claude Agent SDK and lean on its primitives (cwd, env, session resume); we do not re-invent what it already does. The hard parts — driving the claude CLI, persisting the transcript, gating tools — are the SDK's job. This keeps our surface small and our confidence high.

Caveat: the capabilities are real but option names evolve. A real implementation pins a specific SDK version and verifies the surface against it. Treat names here as the capability, not the contract.

02

The decision

We will

Deploy Gavagai as a single internal web app, behind the org's own authentication, running one org API key, giving each user an OS-isolated workspace inside one container — built on the Agent SDK.

The model is org-internal, not bring-your-own-key and not public SaaS. That single framing dissolves the trust problem rather than mitigating it, and it is the only model the SDK actually supports (API-key auth, no third-party login).

D1 · Model

Org-internal

The org runs it inside its own perimeter. The operator is the trusted party — using it asks for no trust beyond email or the corporate drive.

D2 · Auth

Delegate to a reverse proxy

We do not build login. An identity-aware proxy (SSO / oauth2-proxy / VPN) authenticates; Gavagai trusts a signed X-Forwarded-User header.

D3 · Key

One org key at boot

A single ANTHROPIC_API_KEY as platform configuration. No gear, no per-user key, nothing to race over across sessions.

D4 · Persistence

Index + bytes + scratch

A metadata index and a durable byte store wrap a still-real per-session scratch dir (the SDK's cwd), rehydrated with resume.

D5 · Isolation

Per-session OS user, single container, Bash kept

Each user's claude process runs as a distinct unix user with filesystem permissions scoped to its own workspace. The kernel enforces the boundary, so the agent keeps its shell (it needs one to run .tsn) without being able to reach another user's files. No per-session containers, no orchestrator.

03

System shape

One container, fronted by the org's proxy, fed one key as platform config. Inside, the HTTP/SSE server spawns one SDK-driven claude process per connected user, each pinned to its own workspace. Durable state lives outside the container so it survives a restart.

04

Persistence — a layer over a still-real filesystem

The agent needs a real filesystem. Its whole job is to shell out — npx tsx scripts/run.ts, write .tsn files, read PDF bytes. You cannot hand it a database row; it runs ls and cat. So a database does not replace the workspace — it wraps the filesystem, supplying the three things a bare, ephemeral disk cannot: ownership, durability, and history.

The UI endpoints change source, not shape: /files reads the listing from the index (scoped to the caller) instead of readdir; /upload writes to the byte store plus an index row. The chat loop and the file viewer are untouched — same {dir, name, size} JSON.

PoC-scale simplification

At ≤10 users the two outer tiers need not be heavyweight: a persistent volume + SQLite has the same shape as "object store + Postgres" and is plenty. The design is the three roles (index / bytes / scratch); the backends are a deployment knob you can turn up later without changing the shape.

05

Isolation — the heart of the decision

Each connected user spawns its own claude process — that part is easy. The trap is assuming the SDK's cwd option isolates them. It does not. Toggle the diagram to see why, and the fix we chose.

Why this, and why Bash stays

The agent's job is to author and run .tsn — a shell is part of the work, not a risk to design away. We refuse to remove it. So the boundary has to live below the agent, in the OS: spawn each user's process as a distinct unix user whose permissions reach only its own workspace. The agent can ask for anything; the kernel says no. The read-only runtime mount stays shared and unchanged (read-only for everyone — no tenancy concern there).

The payoff is that the whole deployment stays one container — no per-session containers, no scheduler, no cloud-specific machinery — while the isolation is OS-enforced rather than guardrails we write and hope hold.

The one piece of real work

Controlling how the claude subprocess is spawned: it must run under the session's uid/gid. The SDK normally spawns it for us, so the question to settle against the pinned version is whether it exposes spawn uid/gid (or an exec hook), or whether we point it at a small privilege-dropping launcher of our own. That — not cwd — is where the isolation effort sits. Everything else (scratch dir, resume, sync) is unchanged.

06

Build order

Steps 1–2 make it deployable internally for one shared workspace almost immediately. Steps 3–5 are the multi-user build — each leaning on an SDK primitive or a plain OS mechanism, not new infrastructure.

1
Reverse-proxy auth ships v0

Put the proxy in front; trust the identity header. No app code beyond reading it — the SDK has no user-auth, so this layer is required regardless.

2
Org key as platform secret ships v0

Already how it works — sourced from the host's secret store at boot. With (1) this is a usable single-workspace internal tool.

3
Persistence layer

Index + byte store (volume + SQLite at PoC scale); switch /files and /upload to read/write through it; materialize / sync the scratch dir and the session JSONL; resume with the SDK's resume.

4
Per-session OS-user isolation

Spawn each user's claude under a distinct unix user scoped to /ws/<user>. Stays single-container; Bash kept. The uid/gid spawn question (above) is resolved here.

5
Per-user history in the UI

Re-introduce a real "Recents", read from the SDK's own persisted transcript (synced via tier 2) — not a parallel store we maintain.

07

Consequences

What we gain

  • The trust problem dissolves — no one shares a personal key; the operator is the trusted org.
  • Small surface: the SDK does the heavy lifting (CLI, transcript, resume).
  • Single-container operations — no orchestrator, no autoscaling, no cloud lock-in.
  • OS-enforced isolation, not guardrails we have to get right.
  • The container stays disposable: durable state outside means cheap restart / rehydrate.
  • Bash — and the agent's full authoring power — is kept intact.

What it costs / defers

  • Per-user uid/gid spawning is the real engineering bit; correctness rests on the perms setup.
  • One container is a single point of failure and won't scale horizontally — fine at ≤10 users, not beyond.
  • We own identity handling at the proxy boundary (trusting its header).
  • Deferred: container-per-session for genuinely hostile / untrusted tenants.
  • Deferred: token quotas and per-user accounting (soft monitoring only at PoC).
  • Foreclosed: OAuth / claude.ai-login — the SDK does not offer it to third parties.
08

Assumptions

This design is only as sound as the premises under it. These are the ones we had to make; if any breaks, the corresponding decision should be revisited.

A1
The organization is a trust boundary. Users trust their employer with the documents they feed the tool — the same trust email and the corporate drive already assume. This is what lets the operator hold one key and dissolves the BYOK trust problem.
A2
The insider threat is low-stakes — but worth a hard wall anyway. Users are colleagues behind SSO, not adversaries. So "permission denied between workspaces" is sufficient; we do not need hostile-grade sandboxing (gVisor, per-tenant VMs). We still enforce it at the kernel because it is cheap and removes a whole class of accidents.
A3
Scale is ~10 users, not thousands. One container with one CPU/RAM envelope is enough; no autoscaling, no per-session containers, no load balancer. This assumption directly justifies D5 (single container) and the PoC persistence simplification.
A4
An identity-aware proxy exists or can be stood up. The org can place SSO / oauth2-proxy / a VPN in front and inject a trustworthy signed identity header. We trust that header; we do not re-verify the user's credentials.
A5
The Agent SDK exposes cwd / env / resume, and a way to control spawn uid/gid. The persistence and isolation designs depend on these. The uid/gid path in particular is to be verified against the pinned version — either a native option/exec hook, or we wrap the spawn in a privilege-dropping launcher.
A6
The host is Linux and we can create unix users / set ownership inside the container. The container starts privileged enough at setup to create per-user accounts and chown workspaces, then drops privilege per spawned process.
A7
Durable storage and a writable scratch volume are available. At PoC scale a single persistent volume + SQLite stands in for object store + Postgres. The three roles (index / bytes / scratch) are the contract; the backends are swappable.
A8
Outbound network to the Anthropic API is permitted. The agent is useless without it; the org's egress policy allows the container to reach the API with the org key.
A9
The .tsn runtime is not itself an escape vector. Running a .tsn program executes inside the language runtime (a constrained DSL over a read-only mount). We assume it cannot break out to arbitrary host actions beyond what the unix-user permissions already allow. Worth a dedicated look before trusting untrusted-authored programs.
09

Open questions

Settled enough to build; these are the threads to pull as we do.

Session model

One conversation per user, or many named, resumable ones? Decides whether a workspace belongs to a user or a conversation.

Spawn uid/gid

Native SDK option / exec hook, or our own privilege-dropping launcher? The pivot of step 4.

Sync semantics

Turn-boundary snapshot vs. a filesystem watcher; what counts as an "output" worth persisting.

Reap & reconnect

What is "idle", what happens to an in-flight turn, what the user sees on SSE reconnect.

Usage accounting

One key, many users — does the org want per-user token visibility and soft limits? Cheap via the identity header.

Retention & deletion

Lifecycle of uploads, outputs, transcripts; user-initiated delete; the org's data policy.
